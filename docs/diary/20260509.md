## 2. Root cause của vòng lặp AXE-CONFLICT 5 phút

Đây là bug tồn tại nhiều ngày. peer1 cứ 5 phút một lần nhận được WebSocket inbound từ IPv6 của zen (`2a02:c207:2327:7266::1`), trigger AXE conflict resolution, làm churn toàn bộ topology.

### Điều tra

Nhìn vào logs của zen relay: KHÔNG có `[WS-OPEN]` nào đến peer1 sau khi topology đã settle. Source connection hoàn toàn vô hình với relay process.

Kiểm tra `lsof -i :8420` trên zen.akao.io — phát hiện các process `node .../mcp mcp` với PID 8719, 8750, 9155, 9231, 76860. Tất cả đều chạy từ ngày 6-7 tháng 5, nghĩa là đã chạy âm thầm 2-3 ngày.

### Root cause

Các process `mcp mcp` cũ này (trước khi có fix `pid = hash(hwPub + "/mcp")`) dùng chung hardware identity với relay: `pid = hwPub = 0wTwUprS`. Mỗi instance đều khởi động ZEN đầy đủ với DHT-REFRESH timer 5 phút. Cứ 5 phút, `dht.refresh()` gọi `mesh.hi(peer1_url)` → mở WS mới → peer1 thấy một peer mới với `pid=0wTwUprS` (trùng với relay đang có) → AXE-CONFLICT.

Lý do không thấy trong relay logs: connection xuất phát từ một process hoàn toàn khác (pid 8719, etc.), relay process không biết gì.

### Fix tức thì

Kill tất cả 5 process stale qua SSH. Verify: sau khi kill, peer1 sạch không còn AXE event trong 6+ phút.

## 4. Redesign: MCP là relay node thật sự

Nhưng `dht: false` là band-aid. Vision đúng là: **nhúng relay vào mọi thứ** — mỗi MCP instance nên là một relay node đầy đủ trong mesh, không phải con rối bị cắt DHT.

Thực tế đã có fix đúng từ trước: `zenOpt.pid = await ZEN.hash({ data: hwPub + "/mcp" })` — tức là pid của MCP khác với pid của relay (`hwPub`). Nếu pid khác nhau, AXE không conflict. Vậy `dht: false` là không cần thiết.

**Bỏ `dht: false`**, thêm comment giải thích:

```javascript
if (boundServer) {
  // MCP binds port 8420 = MCP IS the relay. Participate fully in the mesh
  // (DHT, BOOT peers, AXE topology). The unique pid = hash(hwPub + "/mcp")
  // below ensures no AXE conflict with the relay process.
  zenOpt.web = boundServer;
}
```

## 5. Phân tích data duplication và process duplication

Câu hỏi đặt ra: với architecture mới, nếu nhiều Copilot session chạy cùng lúc, có bị duplicate data/process không?

### Case 1 (MCP bind TCP 8420)

Chỉ một process có thể bind TCP port — đảm bảo tự nhiên. Không có duplication.

### Case 2 (relay đang chạy → MCP là client)

Phát hiện quan trọng: **IPC singleton đã được thiết kế đúng từ đầu**.

- Relay gọi `attachMcp(zen, { hwIdentity, ipc: true })` → tạo Unix socket `mcp.sock`
- `lib/mcp.js` detect socket → bridge stdio qua socket → **tất cả session dùng chung 1 ZEN instance (relay)**
- Không tạo ZEN instance mới, không data duplication, không process duplication

Tuy nhiên nếu relay chết và session fallback về standalone, nhiều session sẽ chạy `boot()` với cùng `pid = hash(hwPub + "/mcp")` → AXE conflict giữa các MCP standalone.

**Fix**: case 2 fallback dùng `pid = hash(hwPub + "/mcp/" + process.pid)` — unique per OS process:

```javascript
const pidSuffix = zenOpt.web ? "/mcp" : `/mcp/${process.pid}`;
zenOpt.pid = await ZEN.hash({ data: hwPub + pidSuffix });
```

**Bảng tóm tắt kiến trúc:**

| Scenario | Mechanism | Kết quả |
|---|---|---|
| Relay đang chạy | IPC socket → share relay's ZEN | 1 instance, 0 duplicate |
| Relay không chạy, 1 session | standalone boot() case 1 | MCP IS relay |
| Relay không chạy, N sessions | case 2 + unique pid per process | Coexist, no AXE conflict |

## 8. UDP fast path — peer0↔peer1 không hoạt động

### Điều tra

peer0 không bao giờ log `Fast path for 0kiu2ZCd → peer1`. peer1 log thấy fast path ngay khi restart nhưng biến mất ngay sau đó.

Trace vào `setupUdpForPeer`:

```
peer.wire._socket.remoteAddress = "2605:a140:2285:6253::1"  ← IPv6 native
```

Flow cũ:
1. `udpAddr = remoteAddress` → IPv6 string, truthy → URL fallback bị skip
2. `udpAddr.includes(':')` → `return` early
3. UDP fast path không bao giờ được thiết lập

### Root cause

`script/server.js` chỉ tạo một socket `udp4`. Khi WS connection là native IPv6 (peer0→peer1 qua IPv6), `udpAddr` là IPv6 → code cũ không biết xử lý → bail sớm.

Linux ưu tiên IPv6 (RFC 6724) khi peer có AAAA record — peer1.akao.io có cả A và AAAA nên peer0→peer1 WS luôn dùng IPv6.

### Fix (commit `d3680fa`)

Null out native IPv6 trước khi check URL fallback. Result: code rơi vào URL fallback, extract hostname `peer1.akao.io`, dgram resolve qua DNS → IPv4. Tất cả 6 fast paths hoạt động.

## 9. Dual-stack UDP (udp4 + udp6)

### Lý do

Fix trên hoạt động nhưng không tận dụng IPv6: WS dùng IPv6, UDP lại force xuống IPv4 qua hostname DNS lookup. Mất đi ưu điểm của IPv6 (lower latency, no NAT translation).

Quyết định: tạo 2 sockets — `udpSock4` (udp4) và `udpSock6` (udp6, ipv6Only).

### Kiến trúc dual-stack

```
┌─────────────────────────────────────────────┐
│               script/server.js              │
│                                             │
│  udpSock4  :8421  (udp4)                    │
│  udpSock6  :8421  (udp6, ipv6Only: true)    │
│                                             │
│  setupUdpForPeer(peer):                     │
│    ::ffff:x.x.x.x  → strip → udpSock4      │
│    native IPv6     → udpSock6 (if avail)    │
│    hostname/IPv4   → udpSock4               │
└─────────────────────────────────────────────┘
```

`ipv6Only: true` bắt buộc để udp6 chỉ nhận native IPv6 packets, không nhận IPv4-mapped — tránh conflict với udpSock4 trên cùng port.

### Lỗi API: `setIPv6Only` vs `ipv6Only`

Lần đầu implement dùng `socket.setIPv6Only(true)` (method, added v20.1.0). Lỗi: *"udpSock6.setIPv6Only is not a function"* dù Node.js là v24.15.0.

Sửa: `ipv6Only` là **constructor option** (from v11.1.0), không phải method:

```js
// ❌ setIPv6Only là method riêng, không phổ biến
udpSock6.setIPv6Only(true);

// ✅ ipv6Only là option trong createSocket — v11.1.0+
dgram.createSocket({ type: 'udp6', reuseAddr: true, ipv6Only: true })
```

### Kết quả (commit `e4115a0`)

Tất cả 3 server: `[UDP] Listening on :8421 (v4)` + `[UDP] Listening on :8421 (v6)` ✅

Fast paths production:

```
zen → peer0: (v4)   zen → peer1: (v6)  ← peer1 có AAAA, WS IPv6 → UDP IPv6
peer0 → peer1: (v4) peer0 → zen: (v4)+(v6)
peer1 → peer0: (v6) peer1 → zen: (v4)
```

Graceful degradation: nếu platform không hỗ trợ, `udpSock6 = null`, tất cả peers dùng hostname fallback → v4. Không crash.
