## 2. Refactor MCP: boot/attach pattern

Trước đây `lib/mcp/server.js` gộp mọi thứ vào một hàm `start()` cồng kềnh. Đã tách ra thành 3 hàm rõ ràng:

- **`boot()`** — tạo ZEN instance, load hardware identity, trả về `{ zen, hwIdentity, initialPeers }`
- **`attach(zen, opts)`** — mount toàn bộ MCP logic lên ZEN instance đã có sẵn
- **`start()`** — boot() + attach() cho trường hợp standalone

Lý do: muốn relay (`script/server.js`) có thể gọi `attach(zen, { hwIdentity })` và biến relay trở thành MCP server luôn — không cần process thứ hai.

## 3. Nhúng MCP vào relay qua IPC socket

**Thay đổi kiến trúc quan trọng nhất trong session này.**

Trước: MCP server là một process riêng, kết nối vào relay qua WebSocket peer-to-peer. Phức tạp, có latency, cần quản lý hai process.

Sau: MCP server nhúng trực tiếp vào relay worker qua Unix IPC socket (`/tmp/zen-mcp-{port}.sock`).

Cách hoạt động:
- Khi relay khởi động, nếu chạy trong môi trường Node.js, nó tự gọi `import("./service.js")` (lazy load)
- `lib/service.js` gọi `attach(root, { hwIdentity, ipc: true })`
- MCP server lắng nghe trên Unix domain socket
- `lib/mcp.js` (client bridge) detect IPC socket, kết nối qua đó thay vì stdio
- IDE/agent gọi `npx @akaoio/zen mcp` → bridge tới relay qua IPC

**Ưu điểm:**
- Một process duy nhất: relay IS the MCP server
- Không có WebSocket peer phụ làm rối topology
- Browser peer không load MCP (check `Zen.window`)
- Thiết kế giống Gun/SEA: import zen trên, import mcp dưới

Các files thay đổi chính: `lib/mcp/server.js`, `lib/mcp.js`, `script/server.js`, `lib/service.js`.

```
# peer0.akao.io:

```

Topology chain: zen → peer0 → peer1.

## 7. Cuộc chiến với suitePush (zen.push qua relay chain)

### Vấn đề

`zen.push(pubB, data)` gửi ephemeral DAM relay message từ browserA (kết nối zen) tới browserB (kết nối peer1). Phải đi qua chain zen → peer0 → peer1.

Test pass khi chạy riêng lẻ (`--suite=push`), nhưng fail ~65% khi chạy toàn bộ (`--suite=all`).

### Debug hành trình 1: ZEN phải có graph activity trước

Phát hiện: ZEN không gửi HI message (và do đó không có `peer.pub` được set) cho đến khi có ít nhất một GET/PUT activity. Fix: thêm handshake GET trước khi push:

```js
browserA.get("_push-handshake").once(() => {});
browserB.get("_push-handshake").once(() => {});
await sleep(3000);
```

### Debug hành trình 2: zen client cũ (từ suiteChain) làm nhiễu routing

`suiteChain` tạo `zen` client kết nối relay. Sau khi chain test xong, `zen.off()` chỉ đóng logic của ZEN nhưng không đóng WebSocket. Client cũ còn hanging peer làm relay think có thêm peer với pub cũ. Fix: tạo helper `killZen(z)` — đóng wire trực tiếp trước khi gọi `off()`.

### Debug hành trình 3: WebSocket 68s timeout

Sau suiteChain, relay WebSocket connection (68s inactivity) có thể timeout trước khi suitePush bắt đầu. Fix: thêm **30s keepalive ping** trong `src/websocket.js` (outbound connections):

```js
wire.onopen = function() {
  peer._keepalive = setInterval(() => mesh.ping(peer), 30000);
};
wire.onclose = function() {
  clearInterval(peer._keepalive);
};
```

Commit: `7ca8959`

### Debug hành trình 4: peer.wire = null khi bye

Khi một peer disconnect, `mesh.bye()` xóa peer khỏi `opt.peers` nhưng không clear `peer.wire`. Nếu relay cố route message tới peer này trong cùng event loop tick, nó vẫn thấy `peer.wire` còn và gửi → crash hoặc lost message. Fix: `peer.wire = null` ở đầu `mesh.bye()`.

Commit: `8343195`

### Debug hành trình 5: zen.js bundle vs src/ files

Phát hiện quan trọng nhất trong quá trình debug: **relay không dùng `src/mesh.js` trực tiếp, mà dùng file bundle `zen.js` đã build sẵn**.

Flow: `script/server.js` → `lib/server.js` → `import('../zen.js')` (bundle).

Hệ quả: mọi thay đổi trong `src/` không có tác dụng trên relay cho đến khi chạy `npm run buildZEN` và deploy lại `zen.js`. Đã thêm `[RELAY-DBG]` và `[BYE]` logs vào `src/mesh.js` nhưng chúng không xuất hiện trong journalctl vì relay vẫn dùng bundle cũ.

Sau khi rebuild và deploy, debug logs xuất hiện.

### Debug hành trình 6: AXE PID-sort topology collapse

Từ `[BYE]` logs ngay sau relay restart, phát hiện topology collapse:

```
zen:   drops outbound to peer0 (noRec=true)
peer0: drops BOTH outbound AND inbound to peer1 (noRec=true)
peer1: drops inbound from zen (noRec=true)
```

Sau AXE PID-sort, topology ổn định:
- zen: có inbound từ peer0 và peer1 (qua DNS scan của peer1)
- peer0: **CHỈ có zen** — không có kết nối nào tới peer1
- peer1: có outbound tới zen

AXE PID-sort là cơ chế dedup connection: khi hai relay cùng kết nối tới nhau (A→B và B→A), AXE so sánh `opt.pid` (random 9-char per process) và giữ một trong hai. Vấn đề: với chain topology, peer0 luôn mất kết nối tới peer1 sau AXE.

### Debug hành trình 7: DHT routing chọn sai peer

Từ `[RELAY-DBG]` log tại zen:

```
[RELAY-DBG] to=0YscMXGD ttl=3 next=0bUGKvFX livePeers=[0bUGKvFX(w1),0kiu2ZCd(w1),0kiu2ZCd(w1),0LkvOoeu(w1)]
```

zen có kết nối trực tiếp tới peer1 (`0kiu2ZCd`), nhưng DHT vẫn chọn peer0 (`0bUGKvFX`). Lý do:

1. DHT k-buckets (trong `lib/dht.js`) chỉ add peers có `peer.url` (outbound connections). Peer1 là **inbound** tại zen → không có `peer.url` → **không được add vào k-buckets**.
2. `closest()` trả về chỉ peer0 (entry duy nhất trong k-bucket).
3. Zen gửi relay message tới peer0.
4. Peer0 route về zen (chỉ có zen trong livePeers của peer0).
5. Zen nhận lại với cùng `#` → **dedup drops** → message mất.

Vòng lặp: zen → peer0 → zen (dropped) → message không bao giờ tới peer1/browserB.

### Fix cuối cùng: flood relay messages

Thay vì route single-hop XOR-closest, chuyển sang **flood tới tất cả connected peers** (trừ sender):

```js
// BEFORE:
var next = mesh.route(msg.to, peer);
if (next) { mesh.say(fwd, next); }

// AFTER:
for (var fk in peers) {
  var fp = peers[fk];
  if (fp && fp.pub && fp.wire && fp !== peer) { mesh.say(fwd, fp); }
}
```

**Tại sao flooding an toàn:**
- Dedup bằng `msg["#"]` (preserved qua các hop) ngăn true loops
- TTL decrement giới hạn số hop
- Với 3 relays: zen gửi tới peer0+peer1 đồng thời → peer1 deliver trực tiếp tới browserB → peer0 bounce về zen nhưng zen dedup drop → xong

**Tại sao flooding đúng đắn hơn XOR routing cho ephemeral push:**
- XOR routing phù hợp với Kademlia lookup (tìm data stored at closest node)
- Ephemeral push cần gửi tới node đang có kết nối với recipient, không phải node XOR-closest
- Relay mesh nhỏ (vài chục node), flooding cost là O(n) per hop, acceptable

Kết quả: **14/14 tests pass**.

## Bài học rút ra

1. **Bundle vs source**: relay dùng `zen.js` bundle, không phải `src/`. Mọi debug log trong `src/` vô nghĩa cho đến khi rebuild. Từ nay sẽ check bundle date trước khi debug.

2. **AXE topology là unpredictable**: `opt.pid` random mỗi lần process restart → PID-sort kết quả thay đổi mỗi lần deploy → topology khác nhau mỗi lần. Cần thiết kế routing không phụ thuộc vào topology cố định.

3. **Dedup + single-hop routing = silent message loss**: Khi route sai peer, dedup ngăn retry → message mất không trace. Flooding giải quyết cả hai vấn đề.

4. **Inbound peers invisible to DHT**: k-buckets chỉ track outbound (có URL). Peers kết nối vào mình (inbound) không được xét trong routing. Cần nhớ khi debug routing issues.

5. **IPC > subprocess cho MCP**: embed MCP vào relay process sạch hơn nhiều. Không có WebSocket peer phụ làm nhiễu topology, không cần quản lý process lifecycle.
